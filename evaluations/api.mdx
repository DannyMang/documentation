Reference for `/v1/evals/*`. All user-facing endpoints require Bearer token auth (`Authorization: Bearer vk_...`).

## Endpoints

### Eval definitions

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/evals` | Create an eval (dataset + scorers) |
| `GET` | `/v1/evals` | List your evals |
| `GET` | `/v1/evals/{eval_id}` | Get one |
| `DELETE` | `/v1/evals/{eval_id}` | Delete |

### Runs

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/evals/{eval_id}/runs` | Kick off a run against a model |
| `GET` | `/v1/evals/{eval_id}/runs` | List runs for an eval |
| `GET` | `/v1/evals/{eval_id}/runs/{run_id}` | Get a run |
| `GET` | `/v1/evals/{eval_id}/runs/{run_id}/items` | Per-row scored output |
| `POST` | `/v1/evals/{eval_id}/runs/{run_id}/cancel` | Cancel a queued/running run |
| `DELETE` | `/v1/evals/{eval_id}/runs/{run_id}` | Delete a run |

### Worker callbacks

These exist for the eval worker (Fargate service `veri-eval-worker`) and are NOT for users to call. Auth is per-run `callback_token`.

- `POST /v1/evals/{eval_id}/runs/{run_id}/status`
- `POST /v1/evals/{eval_id}/runs/{run_id}/items` (batch upload)

## POST /v1/evals

Create an eval definition. Reusable across many runs.

**Request:**

```json
{
  "name": "GSM8K accuracy",
  "dataset_id": "ds_abc123",
  "scorers": [
    {"type": "exact_match", "name": "exact", "reference_field": "answer"}
  ],
  "generation_params": {
    "max_tokens": 1024,
    "temperature": 0.0,
    "top_p": 1.0
  }
}
```

| Field | Required | Notes |
| --- | --- | --- |
| `name` | yes | â‰¤256 chars |
| `dataset_id` | yes | Must exist + belong to you |
| `scorers` | yes | Non-empty list. See [Scorers](/evaluations/scorers) for shapes |
| `generation_params` | no | Forwarded to inference; optional run-level override |

**Response** (`201`): full `EvalResponse`.

## POST /v1/evals/{eval_id}/runs

Kick off a run.

**Request:**

```json
{
  "model": "deployment:dep_abc123",
  "num_samples": 100,
  "generation_params": {"temperature": 0.7}
}
```

| Field | Required | Notes |
| --- | --- | --- |
| `model` | yes | Today: only `deployment:{id}` works (must be `serving`). `hf:org/model` and `job:<id>` are roadmap. |
| `num_samples` | no | If set, deterministic random sample (`Random(run_id)` seed) |
| `generation_params` | no | Run-level override; falls back to eval-level if not set |

**Response** (`201`): `EvalRunResponse` with `status="queued"`. The eval worker picks it up within ~2 seconds.

## GET /v1/evals/{eval_id}/runs/{run_id}/items

Per-row results. Cursor pagination via `after` (integer item index).

**Query params:** `limit` (1â€“100), `after` (item index).

**Response:**

```json
{
  "data": [
    {
      "object": "eval_run_item",
      "id": "item_xyz",
      "run_id": "run_abc",
      "item_index": 42,
      "input": "{\"role\": \"user\", \"content\": \"What is 2+2?\"}",
      "output": "The answer is 4.",
      "label": "4",
      "scores": {"exact": 0.0, "mentions": 1.0},
      "latency_ms": 245.3,
      "created_at": "2026-05-12T..."
    }
  ],
  "has_more": true
}
```

`scores.<name>` is `null` when a scorer errored on that item; `latency_ms` is `null` if the inference call itself failed.

## EvalRun shape

```json
{
  "object": "eval_run",
  "id": "run_abc123",
  "eval_id": "eval_xyz",
  "model": "deployment:dep_prod",
  "status": "completed",
  "generation_params": {"max_tokens": 2048, "temperature": 0.7, "top_p": 1.0},
  "num_samples": 100,
  "total_items": 100,
  "processed_items": 100,
  "results": {
    "total_items": 100,
    "scores": {
      "exact": {"mean": 0.82, "median": 1.0, "std": 0.384, "min": 0.0, "max": 1.0, "count": 100, "error_count": 0}
    }
  },
  "error_message": null,
  "started_at": "2026-05-12T10:25:00Z",
  "completed_at": "2026-05-12T10:40:15Z",
  "created_at": "2026-05-12T10:24:30Z",
  "updated_at": "2026-05-12T10:40:15Z"
}
```

`results` populates only when `status="completed"`.

## Lifecycle

```
queued â†’ running â†’ completed
                 â†’ failed
                 â†’ cancelled
```

`processed_items` and `total_items` populate during `running`. Progress callbacks from the eval worker fire roughly every 5 seconds.

If the worker silently dies (no status update for >15 minutes), the run is auto-flagged `failed` with `error_message="Worker stale: no status update in 900s"`.

## Generation params

Cascade order: **run-level > eval-level > defaults**. Defaults are `{"max_tokens": 2048, "temperature": 0.7, "top_p": 1.0}`.

These are forwarded to the deployment's `/chat/completions` call. See [OpenAI compatibility](/deployments/openai-compat) for what's actually honored at the serving layer (today: `temperature` + `max_tokens` only).

## Sampling

When `num_samples` is set, the worker selects rows deterministically: `rng = random.Random(run_id)`. Two runs with the same `run_id` would sample the same rows â€” but `run_id` is unique per run, so two new runs always sample independently.

## Cost

Eval runs **don't have separate billing**. Inference cost flows through the deployment via `DeploymentRequest` rows. See [Cost & pricing](/deployments/cost).

## Worker callbacks (appendix)

For completeness â€” these are not user-facing.

### POST /v1/evals/{eval_id}/runs/{run_id}/status

```json
{
  "callback_token": "...",
  "status": "running",
  "processed_items": 25,
  "total_items": 100
}
```

Idempotent â€” late callbacks merge. Sets `last_status_synced_at` for stale detection.

### POST /v1/evals/{eval_id}/runs/{run_id}/items

```json
{
  "callback_token": "...",
  "items": [
    {
      "item_index": 0,
      "input": "...",
      "output": "...",
      "label": "...",
      "scores": {"exact": 1.0},
      "latency_ms": 234.5
    }
  ]
}
```

Worker batches 25 items at a time.
