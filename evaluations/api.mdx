The Evaluations API creates reusable eval definitions and executes them against serving deployments. Runs progress through `preparing`, `generating`, `scoring`, and `aggregating` phases.

## Run workflow

- `POST /v1/evals/{eval_id}/runs` creates a run with immutable provenance.
- `GET /v1/evals/{eval_id}/runs/{run_id}` returns phase, progress, usage, uncertainty, and results.
- `GET /v1/evals/{eval_id}/runs/{run_id}/items` returns candidate request/response evidence and scores.
- `POST /v1/evals/{eval_id}/runs/{run_id}/rescore` creates an immutable scoring revision from persisted outputs without repeating candidate inference.
- `POST /v1/evals/{eval_id}/runs/{run_id}/calibrate` compares an LLM judge revision with boolean item labels.

Calibration labels accept `true`/`false`, `1`/`0`, `pass`/`fail`, or `yes`/`no`. The response includes accuracy, invalid rate, sample count, and a confusion matrix.
