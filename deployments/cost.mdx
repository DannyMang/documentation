Deployments consume GPU credit while they're running. There's no idle auto-stop today — **if you forget to call `stop`, the deployment keeps billing.**

## Billing cadence

Billing starts after the deployment reaches `serving` and continues while it remains `serving` or `unhealthy`.

When you call `stop`, Veri settles the deployment and stops billing.

## Billing states

| Status | Charging? |
| --- | --- |
| `queued` | No |
| `provisioning` | No |
| `serving` | **Yes** |
| `unhealthy` | **Yes** (transient state during health failure) |
| `stopped` | No |
| `failed` | No |

## Idle-stop warning

There is **no idle auto-stop**. A deployment that's `serving` with zero requests for 24 hours will still bill for 24 hours.

Patterns to avoid surprise bills:

1. **Wrap deployments in try/finally** — guarantees `stop` even if your code errors:
   ```python
   dep = client.deployments.create(...)
   try:
       dep.wait()
       # ...do work...
   finally:
       dep.stop()
   ```

2. **Monitor active deployments** — regularly list deployments and stop anything no longer in use.
3. **Set up the low-balance webhook** — Stripe will email at configurable balance thresholds

## Comparison vs training jobs

Training jobs run for a fixed duration and stop themselves. Deployments run **until you stop them**.

If you only need to evaluate a trained model, consider running an eval (which uses an existing deployment briefly) and stopping the deployment immediately after.
