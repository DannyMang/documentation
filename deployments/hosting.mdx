A deployment is a pod on a GPU you control. Picking the GPU matters: too small and the model OOMs at load; too big and you're burning money for headroom you don't use.

## GPU sizing by model size

Rough sizing for **inference only** (training needs more headroom):

| Model size | Minimum | Comfortable | Notes |
| --- | --- | --- | --- |
| ≤ 1B | A100-80GB ×1 | A100-80GB ×1 | Small models always fit on a single GPU. |
| 3–8B | A100-80GB ×1 | H100 ×1 | A100-80GB works; H100 is faster per request. |
| 14–34B | H100 ×2 | H100 ×4 | Tensor parallel; expect TP2 or TP4. |
| 70B+ | H100 ×4 (quantized) | H100 ×8 | TP8 for full precision; quantized at TP4. |

**Rule of thumb:** Serving uses ~half the GPU you trained on. A 7B model trained on 8×A100 typically serves on 1×A100.

## Supported GPU types

| `gpu_type` | Memory | Common use |
| --- | --- | --- |
| `A100-40GB` | 40 GB | Smallest sizes; available on AWS only today |
| `A100-80GB` | 80 GB | Default sweet spot |
| `L40S` | 48 GB | Cost-efficient for 3–8B models on RunPod / AWS |
| `H100` | 80 GB | ~2x throughput vs A100-80GB |
| `H200` | 141 GB | Largest single-GPU memory; needed for some 70B FP16 |

MI300X is priced (TensorWave) but the serving backend is not yet wired — it's on the roadmap.

## Provider routing

When you submit a deployment without specifying `provider`, Veri picks the cheapest available GPU across the priced backends in `control_plane/pricing.json`:

| Provider | Typical strength |
| --- | --- |
| **Lambda Cloud** | Cheapest A100-80GB today |
| **RunPod** | Often best for A100-80GB / L40S |
| **Prime Intellect** | Broad GPU availability |
| **AWS** | Reliable capacity, premium pricing |

<Note>
  The default serving backend on the Veri control plane today is an in-memory stub used during private beta. Direct provider routing for serving is being rolled out — the GPU types and rates above are the published surface; the live backend assignment may differ during early access. The control plane still bills against the chosen `provider` rate.
</Note>

Force a specific provider:

```python
client.deployments.create(
    model="Qwen/Qwen2.5-0.5B-Instruct",
    source="huggingface",
    name="my-deploy",
    gpu={"gpu_type": "H100", "gpu_count": 1},
    provider="lambda",
)
```

Compare rates before deploying:

<CodeGroup>
```python SDK
client.cost.compare(
    gpu_type="H100",
    gpu_count=1,
    duration_minutes=60,
)
```

```bash CLI
veri gpu compare --type H100 --count 1 --hours 1
```
</CodeGroup>

## Cold start

Time from `POST /v1/deployments` to `serving` depends on the provider's pod-lease time plus the model weight download. **Expected** ranges:

| Source | Typical |
| --- | --- |
| `huggingface`, small model (≤3B) | ~1–3 min |
| `huggingface`, 7–13B | ~3–6 min |
| `huggingface`, 30B+ | ~6–15 min (depends on cache hit) |
| `training_job` (your fine-tune) | ~1–3 min — checkpoint already on Veri-managed S3, no HF download |

These are operational expectations, not contractual. The slowest part is usually the model weight download from HF. Once a deployment is `serving`, individual chat requests have sub-second TTFB on small models.

## Endpoint URL stability

Once a deployment reaches `serving`, its `endpoint_url` doesn't change. Save it to your config — it stays valid until you `stop` the deployment.

## OOM at load

If the model fails to load (OOM during weight init, not a request failure), the deployment transitions to `failed` with `error.message` populated.

```python
dep = client.deployments.get(deployment_id)
if dep.status == "failed":
    print(dep.error)
```

Common fix: bump to a larger GPU (`A100-80GB` → `H100`) or add a second GPU (`gpu_count=2`).
