A deployment is a pod on a GPU you control. Picking the GPU matters: too small and the model OOMs at load; too big and you're burning money for headroom you don't use.

## GPU sizing by model size

Rough sizing for **inference only** (training needs more headroom):

| Model size | Minimum | Comfortable | Notes |
| --- | --- | --- | --- |
| ‚â§ 1B | A100-80GB √ó1 | A100-80GB √ó1 | Small models always fit on a single GPU. |
| 3‚Äì8B | A100-80GB √ó1 | H100-80GB √ó1 | A100-80GB works; H100-80GB is faster per request. |
| 14‚Äì34B | H100-80GB √ó2 | H100-80GB √ó4 | Tensor parallel; expect TP2 or TP4. |
| 70B+ | H100-80GB √ó4 (quantized) | H100-80GB √ó8 | TP8 for full precision; quantized at TP4. |

**Rule of thumb:** Serving uses ~half the GPU you trained on. A 7B model trained on 8√óA100 typically serves on 1√óA100.

## Supported GPU types

| `gpu_type` | Memory | Common use |
| --- | --- | --- |
| `A100-40GB` | 40 GB | Smallest sizes; available on AWS only today |
| `A100-80GB` | 80 GB | Default sweet spot |
| `L40S` | 48 GB | Cost-efficient for 3‚Äì8B models on AWS |
| `H100-80GB` | 80 GB | ~2x throughput vs A100-80GB |
| `H200` | 141 GB | Largest single-GPU memory; needed for some 70B FP16 |

MI300X is priced (TensorWave) but the serving backend is not yet wired ‚Äî it's on the roadmap.

## Provider routing

Deployments run on **AWS**. The `provider` field defaults to `aws`; there are no other providers to select today.

<Note>
  The default serving backend on the Veri control plane today is an in-memory stub used during private beta. Direct provider routing for serving is being rolled out ‚Äî the GPU types and rates above are the published surface; the live backend assignment may differ during early access. The control plane still bills against the chosen `provider` rate.
</Note>

Force a specific provider:

```python
client.deployments.create(
    model="Qwen/Qwen2.5-0.5B-Instruct",
    source="huggingface",
    name="my-deploy",
    gpu={"gpu_type": "H100-80GB", "gpu_count": 1},
    provider="lambda",
)
```

Compare rates before deploying:

<CodeGroup>
```python SDK
client.cost.compare(
    gpu_type="H100-80GB",
    gpu_count=1,
    duration_minutes=60,
)
```

```bash CLI
veri gpu compare --type H100-80GB --count 1 --hours 1
```
</CodeGroup>

## Cold start

Time from `POST /v1/deployments` to `serving` depends on the provider's pod-lease time plus the model weight download. **Expected** ranges:

| Source | Typical |
| --- | --- |
| `huggingface`, small model (‚â§3B) | ~1‚Äì3 min |
| `huggingface`, 7‚Äì13B | ~3‚Äì6 min |
| `huggingface`, 30B+ | ~6‚Äì15 min (depends on cache hit) |
| `training_job` (your fine-tune) | ~1‚Äì3 min ‚Äî checkpoint already on Veri-managed S3, no HF download |

These are operational expectations, not contractual. The slowest part is usually the model weight download from HF. Once a deployment is `serving`, individual chat requests have sub-second TTFB on small models.

## Endpoint URL stability

Once a deployment reaches `serving`, its `endpoint_url` doesn't change. Save it to your config ‚Äî it stays valid until you `stop` the deployment.

## OOM at load

If the model fails to load (OOM during weight init, not a request failure), the deployment transitions to `failed` with `error.message` populated.

```python
dep = client.deployments.get(deployment_id)
if dep.status == "failed":
    print(dep.error)
```

Common fix: bump to a larger GPU (`A100-80GB` ‚Üí `H100-80GB`) or add a second GPU (`gpu_count=2`).
