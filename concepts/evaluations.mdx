title: "Evaluations"

Evaluations let you ask the most important question a fine-tuning platform has to answer: **did my training run make the model better?** You point an eval at a [deployed model](/api-reference/deployments) and a [dataset](/concepts/datasets), pick one or more scorers, and Veri returns per-sample scores plus aggregate stats.

## The eval primitive

An evaluation has three parts:

<Steps>
  <Step title="Dataset">
    A JSONL dataset you've already uploaded. Each row needs a `prompt` (string or chat-messages list) and optionally a `label` field that scorers can compare against.
  </Step>
  <Step title="Scorers">
    One or more rules for grading the model's output. Phase A ships `exact_match` and `contains`; reward-function, LLM-as-judge, custom Python, and benchmark scorers land in Phase B/C.
  </Step>
  <Step title="Model target">
    A SERVING [deployment](/api-reference/deployments). Phase A scores against a live deployment endpoint; later phases will support direct HuggingFace targets and ephemeral deployments of training-job checkpoints.
  </Step>
</Steps>

An **Eval** is the (dataset + scorers) definition you create once. An **EvalRun** is one execution of that definition against a specific model — you can run the same eval against the base model and against your fine-tuned model to see the delta.

## The compare-fine-tune-to-base workflow

```python
from veri_sdk import Client

client = Client()

# 1. Define the eval (reuse across runs)
eval = client.evals.create(
    name="math accuracy",
    dataset_id="ds_math_holdout",
    scorers=[{"type": "exact_match", "name": "em", "reference_field": "label"}],
)

# 2. Score the base model deployment
base_run = client.evals.runs.create(eval.id, model=f"deployment:{base_dep_id}")
base_run = base_run.wait()

# 3. Score the fine-tuned model deployment
ft_run = client.evals.runs.create(eval.id, model=f"deployment:{ft_dep_id}")
ft_run = ft_run.wait()

# 4. Compare
print(f"Base:       {base_run.results['scores']['em']['mean']:.3f}")
print(f"Fine-tuned: {ft_run.results['scores']['em']['mean']:.3f}")
```

## Scorers

### `exact_match`

String equality against a dataset reference field. Case-insensitive by default. Both sides are stripped of whitespace. Numeric references (e.g. `{"label": 42}`) are coerced to strings.

```json
{"type": "exact_match", "name": "em", "reference_field": "label", "case_sensitive": false}
```

Good for: math answers, classification labels, exact-recall facts.

### `contains`

Substring or regex match against the dataset reference. Set `regex: true` to compile the reference value as a Python regex; invalid patterns silently score 0.0 (logged, never crashes the run).

```json
{"type": "contains", "name": "has_keyword", "reference_field": "answer", "regex": false}
```

Good for: keyword presence, format-validation hints, partial matches.

### Coming next

- **`reward_function`** — score with the same Python reward you used to train the model. Tight train-eval loop without rewriting anything.
- **`llm_judge`** — LLM-as-judge with a configurable prompt template.
- **`python`** — arbitrary `def score(input, output, item) -> float`.
- **`benchmark`** — curated public benchmarks (MMLU, GPQA, HumanEval, IFEval, MATH-500, GSM8K, …) via [Inspect AI](https://inspect.aisi.org.uk/).

## Run lifecycle

```
queued → running → completed
                 ↘ failed
                 ↘ cancelled
```

The eval worker polls for queued runs and processes them within a few seconds of creation. Progress callbacks land roughly every 5s while a run is in flight (`processed_items` / `total_items` on the run record). When a run completes, `results` is populated with `mean/median/std/min/max/count/error_count` per scorer.

A run is **idempotent under retry**: sampling is deterministic per run ID, so if a worker dies mid-run the replacement scores the same rows.

## Inference and cost

Every sample is a `POST /v1/deployments/{id}/chat/completions` call against your deployment. This means:

1. **Your deployment must be SERVING** at the moment the run starts (the worker re-checks at execution time).
2. **Inference cost is attributed to the deployment**, not the eval — it shows up as `DeploymentRequest` rows and rolls into the hourly deployment billing tick. No separate per-eval surcharge.
3. **`num_samples` caps cost.** For a 14k-sample MMLU-style holdout, start with `num_samples=200` for a quick preview and only run full when you trust the prompt format.

## What evaluations are not

- Not a benchmark suite (yet) — Phase C adds curated public benchmarks via Inspect AI.
- Not production-traffic monitoring — Veri is a training + serving platform; for live-traffic eval, send your `DeploymentRequest` rows or OpenTelemetry traces to Langfuse / Arize / Braintrust.
- Not multi-turn — single-turn scoring only in Phase A. Multi-turn is blocked on the same upstream stability work as multi-turn training.
