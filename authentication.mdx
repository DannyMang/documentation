## Auth Methods

Veri supports two authentication paths:

### 1. Per-User API Keys (SDK / curl)

Create an API key in the dashboard or via the API. Keys start with `vk_` and are used as Bearer tokens:

```bash
curl https://api.veri.studio/v1/training_jobs \
  -H "Authorization: Bearer vk_your_api_key_here"
```

With the Python SDK:

```python
from veri_sdk import Client

client = Client(api_key="vk_your_api_key_here")
```

Or log in via CLI and the SDK reads credentials automatically:

```bash
veri login --key vk_your_api_key_here
```

```python
from veri_sdk import Client
client = Client()  # reads from ~/.veri/credentials.json
```

### 2. Trusted Proxy (Web Dashboard)

When you use the web dashboard, WorkOS handles sign-in and the Next.js proxy attaches your user identity automatically via the `X-Veri-User-Id` header. You don't need an API key for dashboard usage.

For local development or internal testing:

```bash
curl http://localhost:8000/v1/training_jobs \
  -H "Authorization: Bearer $VERI_API_KEY" \
  -H "X-Veri-User-Id: user_123"
```

## API Key Format

Keys follow the format `vk_<40 hex characters>` (43 chars total):

```
vk_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2
```

- The `vk_` prefix is required — the auth middleware uses it to route to API key authentication
- Only the SHA-256 hash is stored server-side
- The plaintext key is shown once at creation — save it immediately
- Keys can be revoked instantly via the API or dashboard

## User Identity & Data Scoping

Every resource you create (datasets, reward functions, training jobs) is scoped to your user account. You can only see and manage your own resources.

Records with no owner (`user_id = NULL`) are inaccessible to all users.

## Environment Variables

```bash
export VERI_API_KEY="vk_your_api_key_here"
```

## Security Best Practices

<Warning>
Never commit API keys, Stripe credentials, or database connection strings to version control.
</Warning>

- **Use environment variables** — Store tokens in `.env` files excluded from git.
- **Keep the control plane private** — The internal service key should only be used by trusted callers.
- **Rotate keys** — Refresh credentials if they are ever shared in logs or chat.

## Error Responses

Authentication errors return `401 Unauthorized`:

```json
{
  "detail": "Invalid API key"
}
```

| Scenario | Response |
|---|---|
| No Authorization header | 401 — "Invalid API key" |
| Invalid or expired key | 401 — "Invalid API key" |
| Revoked `vk_` key | 401 — "Invalid or revoked API key" |
| Service key without X-Veri-User-Id | 401 — "Missing X-Veri-User-Id header" |
