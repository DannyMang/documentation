---
title: "Evaluations"
description: "Score deployed models against a dataset with reusable scorers"
---

Evaluations let you score a SERVING deployment against a dataset using one or more scorers, then read back per-sample results and aggregate stats.

An **Eval** is a definition (dataset + scorers). An **EvalRun** is one execution of that definition against a model target.

## Scorer types

| Type | Status | Description |
|------|--------|-------------|
| `exact_match` | **Shipped** | String equality between the model output and a reference field on the dataset row. Case-insensitive by default. |
| `contains` | **Shipped** | Substring or regex match. The reference value is matched against the output. |
| `reward_function` | Coming (Phase B) | Reuse an uploaded TRL or Miles reward function as the scorer. |
| `llm_judge` | Coming (Phase B) | LLM-as-judge with a prompt template. |
| `python` | Coming (Phase B) | Arbitrary Python `def score(input, output, item) -> float`. |
| `benchmark` | Coming (Phase C) | Curated public benchmarks (MMLU, GPQA, HumanEval, …) via Inspect AI. |

## Create Evaluation

<ParamField body="name" type="string" required>
  Display name for the eval definition.
</ParamField>

<ParamField body="dataset_id" type="string" required>
  ID of the dataset to score against. Must be owned by you. Phase A: JSONL upload datasets only.
</ParamField>

<ParamField body="scorers" type="array" required>
  One or more scorer configs. See "Scorer types" above for the supported shapes.
</ParamField>

<ParamField body="generation_params" type="object">
  Default inference settings (forwarded to the deployment): `max_tokens`, `temperature`, `top_p`, `stop`. Can be overridden per run.
</ParamField>

### Scorer config shapes

**ExactMatch:**
```json
{ "type": "exact_match", "name": "em", "reference_field": "label", "case_sensitive": false }
```

**Contains (substring or regex):**
```json
{ "type": "contains", "name": "has_42", "reference_field": "label", "regex": false }
```

`case_sensitive` defaults to `false`. `regex` defaults to `false` (substring mode). When `regex` is `true`, the reference value is compiled as a Python regex.

### Request

```bash
curl -X POST https://api.veri.studio/v1/evals \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "math accuracy",
    "dataset_id": "ds_abc123",
    "scorers": [
      {"type": "exact_match", "name": "em", "reference_field": "label"},
      {"type": "contains", "name": "has_42", "reference_field": "label"}
    ]
  }'
```

### Response

```json
{
  "id": "ev_xyz789",
  "object": "eval",
  "name": "math accuracy",
  "dataset_id": "ds_abc123",
  "scorers": [
    {"type": "exact_match", "name": "em", "reference_field": "label", "case_sensitive": false},
    {"type": "contains", "name": "has_42", "reference_field": "label", "regex": false}
  ],
  "generation_params": null,
  "created_at": "2026-05-11T10:00:00Z"
}
```

## Create Eval Run

Run the eval against a specific model target. Returns immediately in `queued` status; the eval worker picks it up within a few seconds.

<ParamField body="model" type="string" required>
  Model target. Phase A supports `deployment:<id>` only — the deployment must already be in `serving` status. Phase B will add `hf:<model>` (HuggingFace direct) and `job:<id>` (an ephemeral deployment of a training-job checkpoint).
</ParamField>

<ParamField body="num_samples" type="integer">
  Cap the number of samples scored. Null = full dataset. Sampling is deterministic (seeded by run ID) so retries score the same rows.
</ParamField>

<ParamField body="generation_params" type="object">
  Override the eval's default generation params for this run only.
</ParamField>

### Request

```bash
curl -X POST https://api.veri.studio/v1/evals/ev_xyz789/runs \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deployment:dep_xxx",
    "num_samples": 200
  }'
```

### Response

```json
{
  "id": "run_abc123",
  "object": "eval_run",
  "eval_id": "ev_xyz789",
  "model": "deployment:dep_xxx",
  "status": "queued",
  "num_samples": 200,
  "total_items": null,
  "processed_items": null,
  "results": null,
  "error_message": null,
  "started_at": null,
  "completed_at": null,
  "created_at": "2026-05-11T10:00:00Z",
  "updated_at": "2026-05-11T10:00:00Z"
}
```

## Get Eval Run

```bash
curl https://api.veri.studio/v1/evals/ev_xyz789/runs/run_abc123 \
  -H "Authorization: Bearer vk_your_api_key"
```

When the run reaches `completed`, the `results` field is populated:

```json
{
  "id": "run_abc123",
  "status": "completed",
  "processed_items": 200,
  "total_items": 200,
  "results": {
    "total_items": 200,
    "scores": {
      "em": {
        "mean": 0.825,
        "median": 1.0,
        "std": 0.38,
        "min": 0.0,
        "max": 1.0,
        "count": 200,
        "error_count": 0
      }
    }
  },
  "completed_at": "2026-05-11T10:04:00Z",
  ...
}
```

`error_count` reflects samples where the scorer raised or the inference call failed. Those samples appear in the items list with `scores: {scorer_name: null}` and are excluded from `mean/median/std`.

## Cancel Eval Run

```bash
curl -X POST https://api.veri.studio/v1/evals/ev_xyz789/runs/run_abc123/cancel \
  -H "Authorization: Bearer vk_your_api_key"
```

Marks the run as `cancelled`. The worker bails on its next status-write cycle.

---

## Cost

Eval inference cost is attributed to the customer's **deployment** — every sample is a `POST /v1/deployments/{id}/chat/completions` call, and each call writes a `DeploymentRequest` row that rolls into the deployment's hourly billing tick. There is no separate per-eval surcharge in Phase A.
