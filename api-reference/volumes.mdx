Volumes are named persistent storage that mounts into training workers at `/mnt/<volume-name>`. See the [Volumes](/volumes) guide for the conceptual overview and pricing.

## Create Volume

<ParamField body="name" type="string" required>
  Volume name. Must match `^[a-zA-Z0-9][a-zA-Z0-9_-]{0,127}$`. Idempotent on `(user, name)` — creating the same name twice returns the existing row with status `200`.
</ParamField>

### Request

```bash
curl -X POST https://api.veri.studio/v1/volumes \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{"name": "my-corpus"}'
```

### Response

```json
{
  "object": "volume",
  "id": "vol_abc123",
  "name": "my-corpus",
  "total_bytes": 0,
  "file_count": 0,
  "created_at": "2026-05-14T12:00:00Z",
  "updated_at": "2026-05-14T12:00:00Z"
}
```

Returns `201 Created` for a new volume, `200 OK` for an idempotent hit.

## Get Volume

```bash
curl https://api.veri.studio/v1/volumes/my-corpus \
  -H "Authorization: Bearer vk_your_api_key"
```

Returns `404` if the volume doesn't exist or belongs to a different user.

---

## Delete Volume

```bash
curl -X DELETE https://api.veri.studio/v1/volumes/my-corpus \
  -H "Authorization: Bearer vk_your_api_key"
```

Returns `204 No Content`. The volume row is dropped synchronously; the S3 prefix is reaped by a background reconciler within ~60 seconds. Idempotent — deleting a non-existent volume is also `204`.

## Upload Files

The upload protocol is two-step:

1. `POST /v1/volumes/{name}/upload-url` with the file's size. The response tells you whether to do a single-shot PUT or a multipart upload.
2. PUT the file (or each part) directly to the presigned S3 URL the server returned. The bytes never traverse the Veri control plane.

### Request a presigned upload URL

<ParamField body="path" type="string" required>
  Remote path inside the volume. Must start with `/`. No `..` segments.
</ParamField>

<ParamField body="size_bytes" type="integer" required>
  File size in bytes. Files larger than 5 GiB trigger a multipart upload response.
</ParamField>

```bash
curl -X POST https://api.veri.studio/v1/volumes/my-corpus/upload-url \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{"path": "/train.jsonl", "size_bytes": 1500000}'
```

For files ≤ 5 GiB the response is a single-shot PUT URL:

```json
{
  "type": "single",
  "url": "https://veri-volumes.s3.amazonaws.com/volumes/vol_abc123/train.jsonl?X-Amz-..."
}
```

PUT the file to that URL with standard HTTP. Authentication is baked into the URL — do **not** pass your API key on the PUT.

For files > 5 GiB the response initiates a multipart upload:

```json
{
  "type": "multipart_init",
  "upload_id": "abc-multipart-id",
  "path": "/big.bin",
  "part_size": 67108864,
  "complete_url": "/v1/volumes/my-corpus/upload-url/complete"
}
```

### Request a per-part URL

For each part (1-indexed, max 10,000):

```bash
curl -X POST https://api.veri.studio/v1/volumes/my-corpus/upload-url/part \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "/big.bin",
    "upload_id": "abc-multipart-id",
    "part_number": 1
  }'
```

```json
{ "url": "https://veri-volumes.s3.amazonaws.com/...&partNumber=1" }
```

PUT the part bytes to that URL. The S3 response includes an `ETag` header — collect it for the completion call.

### Complete the multipart upload

```bash
curl -X POST https://api.veri.studio/v1/volumes/my-corpus/upload-url/complete \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "/big.bin",
    "upload_id": "abc-multipart-id",
    "parts": [
      {"number": 1, "etag": "abc..."},
      {"number": 2, "etag": "def..."}
    ]
  }'
```

The server completes the multipart upload, HEADs the final object for size + ETag, and atomically updates the volume's `total_bytes` and `file_count`:

```json
{
  "object": "volume.file",
  "path": "/big.bin",
  "size_bytes": 6442450944,
  "etag": "merged-etag"
}
```

<Note>
  Abandoned multipart uploads are reaped by an S3 lifecycle rule after 7 days — that rule is the load-bearing cleanup. The Python SDK also best-effort-aborts on any part failure (no separate HTTP endpoint), but you can rely on the lifecycle rule even if the SDK crashes mid-upload.
</Note>

## Delete File

```bash
curl -X DELETE https://api.veri.studio/v1/volumes/my-corpus/files/data/old.jsonl \
  -H "Authorization: Bearer vk_your_api_key"
```

Returns `204 No Content`. Idempotent — deleting a non-existent file is also `204`. `total_bytes` and `file_count` on the volume are decremented atomically.

## Using a Volume in a training job

Attach a volume to a training job by referencing it through a dataset. Create a `Dataset` with `source_type="volume"`:

```bash
curl -X POST https://api.veri.studio/v1/datasets/connect \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "corpus-train",
    "source_type": "volume",
    "volume": "my-corpus",
    "volume_path": "/train.jsonl"
  }'
```

Then submit the training job using that `dataset_id`. The worker mounts `my-corpus` at `/mnt/my-corpus`, and your script reads `/mnt/my-corpus/train.jsonl` directly — no in-memory list materialization.

You can also pass `volume` directly on `POST /v1/training_jobs` for an explicit mount; either path produces the same worker config.

## Error reference

| Status | Body | Cause |
| --- | --- | --- |
| `400 invalid_request_error` | `volume_path must start with '/'` | Path validation failed. |
| `404 not_found_error` | `Volume not found` | Volume doesn't exist or belongs to another user. |
| `409 invalid_request_error` | `dataset_name_conflict` | Reusing a dataset name for a different file in the same volume. |
| `422 unprocessable_entity` | Pydantic validation error | Name regex / required field violations. |
