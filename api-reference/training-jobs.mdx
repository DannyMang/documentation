## Create Training Job

<ParamField body="base_model" type="string" required>
  HuggingFace model ID (e.g., `Qwen/Qwen3-4B`, `THUDM/CogVideoX-2b`).
</ParamField>

<ParamField body="dataset_id" type="string" required>
  ID of the uploaded dataset to train on.
</ParamField>

<ParamField body="reward_function_id" type="string">
  ID of the uploaded reward function. Required for `grpo`. Must be omitted for `sft_video_gen`.
</ParamField>

<ParamField body="method" type="string" default="grpo">
  Training method: `grpo` or `sft_video_gen`.
</ParamField>

<ParamField body="output_name" type="string" required>
  Name for the output model checkpoint.
</ParamField>

<ParamField body="hyperparameters" type="object">
  Training hyperparameters. The schema depends on the `method` field.

  <Expandable title="GRPO hyperparameters (method: grpo)">
    <ParamField body="learning_rate" type="float" default="1e-6">
      Learning rate for the optimizer.
    </ParamField>
    <ParamField body="num_epochs" type="integer" default="1">
      Number of training epochs.
    </ParamField>
    <ParamField body="max_steps" type="integer">
      Optional explicit training step limit.
    </ParamField>
    <ParamField body="rollouts_per_prompt" type="integer" default="8">
      Number of completions generated per prompt.
    </ParamField>
    <ParamField body="kl_coef" type="float" default="0.001">
      KL penalty coefficient.
    </ParamField>
    <ParamField body="max_prompt_length" type="integer" default="1024">
      Prompt token cap.
    </ParamField>
    <ParamField body="max_response_length" type="integer" default="2048">
      Completion token cap.
    </ParamField>
    <ParamField body="global_batch_size" type="integer" default="64">
      Global batch size.
    </ParamField>
    <ParamField body="seed" type="integer" default="42">
      Random seed.
    </ParamField>
    <ParamField body="use_unsloth" type="boolean" default="true">
      Use the Unsloth optimized GRPO trainer.
    </ParamField>
    <ParamField body="lora_rank" type="integer | null" default="null">
      Must be `null`. LoRA fine-tuning is not yet supported for GRPO — the API accepts the field but the trainer requires `null`.
    </ParamField>
    <ParamField body="lora_alpha" type="integer | null" default="null">
      Must be `null`. See `lora_rank` above.
    </ParamField>
  </Expandable>

  <Expandable title="Agentic GRPO hyperparameters (method: grpo_agentic)">
    Inherits every GRPO field above. Adds:

    <ParamField body="max_turns" type="integer" default="5">
      Maximum number of agent turns per rollout.
    </ParamField>
    <ParamField body="max_seq_len" type="integer" default="8192">
      Maximum total sequence length across all turns.
    </ParamField>

    <Note>
      `grpo_agentic` requires H100 GPUs and the `environments` reward source (mutually exclusive with `reward_function_id`).
    </Note>
  </Expandable>

  <Expandable title="Video Gen SFT hyperparameters (method: sft_video_gen)">
    <ParamField body="learning_rate" type="float" default="1e-3">
      Learning rate for the optimizer.
    </ParamField>
    <ParamField body="num_epochs" type="integer" default="30">
      Number of training epochs.
    </ParamField>
    <ParamField body="max_steps" type="integer">
      Optional explicit training step limit.
    </ParamField>
    <ParamField body="lora_rank" type="integer" default="64">
      LoRA adapter rank.
    </ParamField>
    <ParamField body="lora_alpha" type="integer" default="64">
      LoRA scaling factor.
    </ParamField>
    <ParamField body="resolution_height" type="integer" default="480">
      Video frame height in pixels.
    </ParamField>
    <ParamField body="resolution_width" type="integer" default="720">
      Video frame width in pixels.
    </ParamField>
    <ParamField body="num_frames" type="integer" default="49">
      Number of video frames per sample.
    </ParamField>
    <ParamField body="fps" type="integer" default="8">
      Frames per second.
    </ParamField>
    <ParamField body="batch_size" type="integer" default="1">
      Per-device batch size.
    </ParamField>
    <ParamField body="gradient_accumulation_steps" type="integer" default="4">
      Gradient accumulation steps.
    </ParamField>
    <ParamField body="seed" type="integer" default="42">
      Random seed.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="gpu" type="object" required>
  Explicit GPU configuration. Object with `gpu_type` (one of `A100-40GB`, `A100-80GB`, `H100`, `L40S`) and `gpu_count` (1–8).
</ParamField>

<ParamField body="provider" type="string">
  Compute provider: `prime_intellect`, `lambda`, `runpod`. Omit to auto-select the cheapest available.
</ParamField>

<ParamField body="checkpoint_destination" type="object">
  Optional final checkpoint target. Defaults to `{ "type": "veri" }`.
</ParamField>

<ParamField body="environments" type="object">
  OpenReward environments to use as the reward signal. Mutually exclusive with `reward_function_id` — pick one reward source. Each key is an environment ID, and the value contains configuration for that environment.

  ```json
  {
    "coding_sandbox": { "splits": ["train"] }
  }
  ```

  <Note>
  Requires OpenReward integration to be configured first via `PUT /v1/settings/integrations/openreward`.
  </Note>
</ParamField>

### Request (GRPO)

```bash
curl -X POST https://api.veri.studio/v1/training_jobs \
  -H "Authorization: Bearer vk_your_api_key" \
  -H "Content-Type: application/json" \
  -d '{
    "base_model": "Qwen/Qwen3-4B",
    "dataset_id": "ds_abc123",
    "reward_function_id": "rf_def456",
    "method": "grpo",
    "output_name": "qwen3-4b-math",
    "hyperparameters": {
      "learning_rate": 1e-6,
      "num_epochs": 1,
      "rollouts_per_prompt": 8,
      "max_response_length": 2048,
      "global_batch_size": 64
    },
    "gpu": {
      "gpu_type": "A100-80GB",
      "gpu_count": 1
    },
    "checkpoint_destination": {
      "type": "veri"
    }
  }'
```

    "dataset_id": "ds_abc123",

    "output_name": "qwen3-4b-coding",

### Job Statuses

| Status | Description |
|--------|-------------|
| `queued` | Job row created and waiting to be submitted |
| `provisioning` | Backend accepted the job and compute is starting |
| `running` | Training is in progress |
| `completed` | Training finished successfully; checkpoint is available |
| `failed` | Training failed; check `error.message` and `error.code` |
| `cancelled` | Job was cancelled by the user |

## Get Training Metrics

Returns sampled per-step training metrics (loss, reward, KL, learning rate, and any other numeric value emitted by the trainer). Metrics are sampled at ~5 second intervals — the worker throttles status callbacks, so adjacent steps within one window collapse to the most recent.

<ParamField query="keys" type="string">
  Comma-separated list of metric keys to return (e.g. `loss,reward`). When omitted, all available keys are returned.
</ParamField>

<ParamField query="from_step" type="integer">
  Lower bound (inclusive). Pass `latest_step + 1` from the previous response for incremental polling.
</ParamField>

<ParamField query="to_step" type="integer">
  Upper bound (inclusive).
</ParamField>

<ParamField query="max_points" type="integer" default="500">
  Server-side stride downsample target. Long runs are reduced to at most this many evenly-spaced points.
</ParamField>

### Request

```bash
curl "https://api.veri.studio/v1/training_jobs/job_xyz789/metrics?keys=loss,reward&max_points=500" \
  -H "Authorization: Bearer vk_your_api_key"
```

### Response

```json
{
  "metrics": [
    { "step": 0, "values": { "loss": 1.42, "reward": 0.10 } },
    { "step": 5, "values": { "loss": 1.31, "reward": 0.18 } },
    { "step": 10, "values": { "loss": 1.20, "reward": 0.25 } }
  ],
  "available_keys": ["loss", "reward", "kl", "learning_rate"],
  "latest_step": 1234
}
```

`available_keys` always reflects every metric key persisted for the job, regardless of `keys` filtering. `latest_step` is the absolute maximum step regardless of any stride downsample, so clients can use it as the next `from_step - 1` for polling.

## Cancel Training Job

### Request

```bash
curl -X POST https://api.veri.studio/v1/training_jobs/job_xyz789/cancel \
  -H "Authorization: Bearer vk_your_api_key"
```

### Response

```json
{
  "id": "job_xyz789",
  "object": "training_job",
  "status": "cancelled"
}
```

<Note>
Only non-terminal jobs can be cancelled. If the job is already `completed`, `failed`, or `cancelled`, the API returns `400`.
</Note>

## Stream Job Logs

Returns job logs as Server-Sent Events (SSE).

<Note>
Real-time log streaming from training is not yet implemented. Currently this endpoint returns a link to the provider dashboard where logs can be viewed. Full SSE streaming of training progress (step, loss, reward) is planned.
</Note>

### Request

```bash
curl -N https://api.veri.studio/v1/training_jobs/job_xyz789/logs \
  -H "Authorization: Bearer vk_your_api_key"
```

### Response (SSE stream)

The endpoint emits an SSE stream of `data:` lines while the job is still running. When the runner has not yet attached or only bootstrap output is available, the stream falls back to a one-shot `bootstrap.log` + `worker_status.log_tail` payload. For completed runs, prefer `GET /v1/training_jobs/{id}/logs/full` instead.

## Get Full Logs

Returns the full plain-text log for a finished (or partially finished) job. Useful when SSE has dropped or for post-mortems.

### Request

```bash
curl https://api.veri.studio/v1/training_jobs/job_xyz789/logs/full \
  -H "Authorization: Bearer vk_your_api_key"
```

Response: `200 OK` with `Content-Type: text/plain` and the log body. Returns `206 Partial Content` if the log is still streaming (more lines will follow once the run finishes), or `404 Not Found` if no log exists yet.

## Heartbeat

Worker liveness signal. Called by the runner — clients generally do not invoke this.

### Request

```bash
curl -X POST https://api.veri.studio/v1/training_jobs/job_xyz789/heartbeat \
  -H "Authorization: Bearer vk_your_api_key"
```

Returns `204 No Content`. Used by the SDK's job-tracking helpers to detect stuck workers.

## Delete Training Job

```bash
curl -X DELETE https://api.veri.studio/v1/training_jobs/job_xyz789 \
  -H "Authorization: Bearer vk_your_api_key"
```

Returns `204 No Content`. Only valid for jobs in terminal states (`completed`, `failed`, `cancelled`) **and** with no active deployments referencing them.

## Get Model Checkpoint

Returns a download URL for the trained model checkpoint. Only available for jobs with `completed` status.

### Request

```bash
curl https://api.veri.studio/v1/training_jobs/job_xyz789/model \
  -H "Authorization: Bearer vk_your_api_key"
```

### Response

```json
{
  "download_url": "https://storage.veri.studio/checkpoints/job_xyz789/model.tar.gz"
}
```

<Note>
The same URL is also surfaced as `download_url` on completed job responses when checkpoint resolution succeeds.
</Note>
